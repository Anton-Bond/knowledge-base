#### 1. **Ресурсы**

- В REST API всё — это ресурс.
- Используйте множественное число для названия своих REST сервисов. Если вы придерживаетесь единственного числа, то придерживайтесь этого во всех своих сервисах. Не смешивайте использование множественного и единственного чисел.
    - **Неравильно**: `/user`, `/product`
    - **Правильно**: `/users`, `/products`

---

#### 2. **Уникальные идентификаторы**

- Каждый ресурс должен иметь уникальный идентификатор. Например:
    - `/users/123` — где `123` — это идентификатор ресурса "пользователь".
- Используйте значения, которые легко интерпретируются, например UUID, вместо сложных числовых последовательностей.

---

#### 3. **Стандартизация**

- Используйте стандартные методы HTTP для выполнения операций:
    - **GET** — для получения данных.
    - **POST** — для создания данных.
    - **PUT** — для полного обновления.
    - **PATCH** — для частичного обновления.
    - **DELETE** — для удаления.

---

#### 4. **Чистые и лаконичные URL**

- Используйте понятные и читаемые URL.
    - **Правильно**:
        
        ```plaintext
        GET /users
        GET /users/123
        GET /users/123/orders
        ```
        
    - **Неправильно**:
        
        ```plaintext
        GET /getUsers
        GET /123-user
        GET /usersOrders/123
        ```
        
- Избегайте использования глаголов в URL, действия задаются HTTP-методами.

---

#### 5. **Использование HTTP-статусов**

- Возвращайте соответствующий HTTP-статус для каждого запроса:
    - Успех: `200 OK`, `201 Created`, `204 No Content`.
    - Ошибка клиента: `400 Bad Request`, `401 Unauthorized`, `404 Not Found`, `409 Conflict`.
    - Ошибка сервера: `500 Internal Server Error`, `503 Service Unavailable`.

---

#### 6. **Документация**

- Всегда документируйте ваш API.
- Используйте инструменты вроде **Swagger** или **OpenAPI** для создания удобной документации. Это поможет разработчикам легко понять и работать с вашим API.

---

#### 7. **Безопасность**

- Используйте **аутентификацию** и **авторизацию**.
    - JWT (JSON Web Token) — современный подход для защиты API.
    - OAuth2 — для более сложных систем.
- Передавайте чувствительные данные только через **HTTPS**.

---

#### 8. **Обработка ошибок**

- Возвращайте понятные сообщения об ошибках:
    
    ```json
    {
      "error": "User not found",
      "status": 404
    }
    ```
    
- Следуйте стандартам: указывайте коды ошибок и причины.

---

#### 9. **Кэширование**

- Для высоконагруженных систем используйте кэширование для снижения нагрузки.
    - HTTP-заголовки:
        - `Cache-Control`
        - `ETag`
    - Системы кэширования: Redis, Memcached.

---

#### 10. **Идемпотентность**

- Идемпотентные методы: **GET**, **PUT**, **DELETE** — несколько вызовов не изменят результат.
- Неидемпотентные методы: **POST**, **PATCH**. Например, POST создает новый ресурс, и повторный вызов создаст еще один.

---

#### 11. **Неидемпотентные методы**

- **POST**: Создание нового ресурса.
- **PATCH**: Изменение существующего ресурса (неидемпотентный, так как изменения могут быть разные при каждом вызове).

---

#### 12. **Другие принципы**

- **Версионирование**: Добавляйте номер версии в URL, чтобы избежать проблем при изменениях. Например:
    
    ```plaintext
    GET /v1/users
    ```
    
- **HATEOAS** (Hypermedia As The Engine Of Application State): Возвращайте ссылки на доступные действия.
