### 1. **Что такое REST API?**

**REST** (Representational State Transfer) — это архитектурный стиль, используемый для создания веб-сервисов. REST API предоставляет интерфейс для взаимодействия между клиентом и сервером через протокол HTTP, опираясь на ряд принципов:

- **Ресурсы**: В REST всё — это ресурс. Ресурс может быть любым объектом, таким как пользователь, файл или заказ.
- **Методы HTTP**: Используются стандартные методы (GET, POST, PUT, DELETE и другие) для выполнения операций над ресурсами.
- **URI (Uniform Resource Identifier)**: Каждый ресурс идентифицируется уникальным URL.
- **Состояние без сохранения**: Каждый запрос клиента к серверу полностью независим, сервер не хранит состояние сессии.
- **Кэширование**: Рекомендуется использовать кэш для ускорения работы.

---

### 2. **Разница между REST API и RESTful API**

- **REST API** — это обобщенный термин, который может использоваться для обозначения любого API, соответствующего части REST-принципов. Например, может быть реализовано только использование методов HTTP, но без строгого соблюдения всех принципов REST.
- **RESTful API** — это API, которое строго следует всем принципам REST. Это своего рода "идеальная" реализация REST.

Ключевая разница: REST API может нарушать некоторые REST-принципы, тогда как RESTful API обязательно им следует.

---

### 3. **Основные методы HTTP**

В REST API используются стандартные методы HTTP:

- **GET**: Чтение ресурса (идемпотентный).
- **POST**: Создание ресурса (неидемпотентный).
- **PUT**: Полное обновление ресурса (идемпотентный).
- **PATCH**: Частичное обновление ресурса (неидемпотентный).
- **DELETE**: Удаление ресурса (идемпотентный).

---

### 4. **Разница между PUT и PATCH**

- **PUT**: Используется для полного обновления ресурса. Если ресурса нет, он создается.  
    Пример:
    
    ```
    PUT /users/123
    {
      "name": "John",
      "email": "john@example.com"
    }
    ```
    
    **Результат**: Все поля ресурса заменяются.
    
- **PATCH**: Используется для частичного обновления ресурса.  
    Пример:
    
    ```
    PATCH /users/123
    {
      "email": "john.new@example.com"
    }
    ```
    
    **Результат**: Обновляется только указанное поле, остальные остаются без изменений.
    

---

### 5. **HTTP Статусы**

HTTP-статусы помогают сообщать клиенту о результате обработки запроса:

- **2xx (Успех)**:
    - `200 OK` — Запрос успешен.
    - `201 Created` — Ресурс создан.
    - `204 No Content` — Успешно, но без данных в ответе.
- **3xx (Перенаправления)**:
    - `301 Moved Permanently` — Ресурс перемещен навсегда.
    - `304 Not Modified` — Кэшированные данные не изменились.
- **4xx (Ошибка клиента)**:
    - `400 Bad Request` — Ошибка в запросе.
    - `401 Unauthorized` — Требуется авторизация.
    - `404 Not Found` — Ресурс не найден.
    - `409 Conflict` — Конфликт, например, при создании уже существующего ресурса.
- **5xx (Ошибка сервера)**:
    - `500 Internal Server Error` — Общая ошибка сервера.
    - `503 Service Unavailable` — Сервис временно недоступен.
